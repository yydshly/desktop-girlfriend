# 桌面女友呈现层架构

这份文档解释当前 `showcase-demo` 的真实定位：它不是视频播放器，也不是素材仓库，而是“桌面女友视觉呈现层”的原型。

## 核心分层

```text
AI / 状态层
  -> 负责对话、情绪、意图、当前状态
  -> 例如：happy、thinking、speaking、idle、lookAtCursor

动作调度层
  -> 把状态翻译成视觉表现
  -> 例如：happy 触发暖色粒子；speaking 触发口型；lookAtCursor 触发视线跟随

视觉渲染层
  -> 负责真正显示出来
  -> 当前：人物图片 + CSS 动画 + Canvas 粒子
  -> 未来：Live2D / Spine / Rive / Three.js
```

当前 `showcase-demo` 主要实现后两层：

- 动作调度层：`dispatchCompanionAction(...)`
- 视觉渲染层：人物图、粒子、眼神追踪、口型占位、状态按钮、事件日志

## 为什么第一版使用人物图片

你现在没有视频生成模型，也不需要为了第一版 demo 依赖视频。

更合理的第一版是：

```text
一张稳定的人物参考图
  + 前端动画
  + 状态事件
  + 粒子和口型占位
```

这样可以先验证真正重要的桌面呈现问题：

- 她是否能稳定出现在桌面场景里
- 状态变化是否能带来可见反馈
- 情绪是否能影响氛围
- 鼠标注视、说话、思考、待机是否能被统一调度
- 后续是否能替换成 Live2D 或 Spine

## 当前两个页面的定位

### `runtime.html`

这是现在最重要的页面。

它展示“AI 状态如何驱动人物表现”：

- 主舞台人物
- 情绪按钮
- 动作按钮
- AI 事件预设
- 当前状态读数
- 事件日志

它更接近未来桌面女友的运行时。

### `index.html`

这是展示页。

它展示 6 个能力模块：

- 走路效果
- 手势动作
- 眼神追踪
- 情绪粒子
- 唱歌效果
- 待机互动

它适合用来说明功能方向，但不是最终运行时架构。

## 每一步能看到什么效果

### Step 1：人物素材上屏

素材：

```text
assets/reference/character-reference.png
```

效果：

- 所有主要场景复用同一个人物素材
- 角色形象统一
- 不依赖视频模型

### Step 2：基础动作假驱动

实现：

- CSS 位移
- CSS 轻微旋转
- CSS 呼吸和摆动

效果：

- 待机时轻微呼吸
- 思考时轻微歪头
- 打招呼时轻微摆动
- 唱歌时身体轻摆

### Step 3：实时交互

实现：

- 鼠标位置驱动 `lookAtCursor`
- `dispatchCompanionAction(...)` 驱动动作和情绪

效果：

- 角色可以看向鼠标
- 外部状态可以触发开心、思考、说话、待机等表现

### Step 4：情绪氛围

实现：

- Canvas 粒子
- 不同 emotion preset

效果：

- `happy`：暖色粒子
- `calm`：低密度柔和粒子
- `excited`：高密度亮色粒子
- `sad`：冷色下落粒子

### Step 5：口型占位

实现：

- 前端 mouth layer
- `speaking` 状态控制

效果：

- 不需要真实 TTS，也能先看到“正在说话/唱歌”的视觉提示
- 后续可接音频振幅做真实 lip-sync

## 视频在这里是什么角色

视频不是核心架构，只是可选渲染后端。

默认使用：

```text
人物图片 + 前端动画
```

只有打开下面地址时才启用视频素材：

```text
index.html?renderer=video
```

这样视频不会绑架架构。未来如果你有 Hailuo/MiniMax 之类的视频素材，可以作为某些动作的临时表现；如果要做长期桌面女友，应升级到 Live2D / Spine。
